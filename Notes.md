# Docker vs Podman

- [What is Podman? (redhat.com)](https://www.redhat.com/en/topics/containers/what-is-podman)
- [what is Buildah? (redhat.com)](https://www.redhat.com/en/topics/containers/what-is-buildah)
- [What is Skopeo? (redhat.com)](https://www.redhat.com/en/topics/containers/what-is-skopeo)
- [Tutorials — Podman documentation](https://docs.podman.io/en/latest/Tutorials.html)

* Docker defaults to individual containers, while Podman uses pods of N>=1 containers.
* Docker is all-in-one, Podman uses separate tools.
* Docker-build vs Podman-build, plus
	* Buildah and Skopeh (resp. build and move images).
	* Podman-build is a subset of Buildah functionality
	* Skopeo allows you to copy images between repos and to sync repos among themselves.
* Docker's daemon runs as root, Podman has no daemon and it runs as non-root.
	* Because of this, each user on a system has their own set of images, repositories, volumes and containers.
* Docker **was** not compatible with CRI, while Podman, CRI-O and Kubernetes were.
* Docker's containers use namespaces etc, while Podman also assigns unique SELinux contexts.
* Docker uses its own daemon, while Podman integrates with Systemd.

Plus:

* Both adhere to OCI.
* Both use the same commands; you can _alias docker=podman._
* Containers built on one, work on the other and both on Kubernetes.

# Docker Engine

In early days, the Docker daemon used LXC to interact with the kernel to create namespaces, CGroups etc. Then they have replaced it with libcontainer to directly interact with the kernel.

These days, Docker consists of multiple parts:

* _containerd_, to manage the containers
* _runc_, to actually run containers, with alternatives like _crun_, _vrun_ and so on.
* _dockerd_, to pull and manage images, plus volumes and networks.

## Important note about bind mount and SELinux

You may have to bind-mount a directory from the host OS into the container. For example when you mount a web htdocs directory externally.

On systems that have SELinux enabled, this will need the following commands to set the file context for the directory to _container\_file\_t._

```
sudo semanage fcontext -a -t container_file_t "/web(/.*)?"
sudo restorecon -R -v /web
```

## Registries in Podman

Configuring Podman to use specific registries is different from Docker. I'd say it's easier than with Docker!

In _/etc/containers/registries.conf_ is where you set everything up. Here's a sample:

```
unqualified-search-registries = ["localhost:5000", "registry.access.redhat.com", "registry.redhat.io", "docker.io"]

[[registry]]
location = "localhost:5000"
insecure = true
```

## Setting up your own registry with Podman

The DO180 sample exam on Github has playbooks which pretty neatly give instructions on how to run your own registry.

The following ensures that the registry is managed by, and started by, the root user.

```
sudo dnf install -y podman
sudo podman pull registry:2
sudo mkdir /var/lib/registry
sudo podman run -d --name registry -p 5000:5000 -v /var/lib/registry:/var/lib/registry:z registry:2
sudo podman generate systemd registry > /etc/systemd/system/registry.service
sudo systemctl daemon-reload
sudo systemctl enable registry.service
sudo systemctl restart registry.service

sudo systemctl enable firewalld
sudo firewall-cmd --add-port 5000/tcp
```

You can also make it run as your own user account:

```
sudo dnf install -y podman
podman pull registry:2
mkdir ~/registry
podman run -d --name registry -p 5000:5000 -v ${HOME}/registry:/var/lib/registry:z registry:2
podman generate systemd registry | sudo tee -a /etc/systemd/user/registry.service
sudo chmod 644 /etc/systemd/user/registry.service
systemctl --user daemon-reload
systemctl --user enable registry.service
systemctl --user restart registry.service

sudo systemctl enable firewalld
sudo firewall-cmd --add-port 5000/tcp
```

Working with variables and arguments:
The `${variable_name}` syntax also supports a few of the standard `bash` modifiers as specified below:

* `${variable:-word}` indicates that if `variable` is set then the result will be that value. If `variable` is not set then `word` will be the result.
* `${variable:+word}` indicates that if `variable` is set then `word` will be the result, otherwise the result is the empty string.

# Podman compose and RHEL

To use _podman-compose_ on RHEL, you will need to enable the EPEL repositories. ([source](https://access.redhat.com/discussions/6979552))

`sudo subscription-manager repos --enable codeready-builder-for-rhel-9-x86_64-rpms`
`sudo dnf install` <https://dl.fedoraproject.org/pub/epel/epel-release-latest-9.noarch.rpm>
`sudo dnf upgrade`
`sudo dnf install podman-compose`

or

```
$ sudo /usr/bin/crb enable
$ sudo dnf install https://dl.fedoraproject.org/pub/epel/epel-release-latest-8.noarch.rpm
$ sudo dnf install podman-compose
```
